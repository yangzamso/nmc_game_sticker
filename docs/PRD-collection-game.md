# PRD: 옷 컬렉션 미니게임 (가챠 캡슐 방식)

## 1. 배경 및 목적

기존 코디(옷 갈아입기)/인쇄 게임([GameBoard.jsx](../src/components/GameBoard.jsx))은 유지한 채, 그 앞단계로 "미니게임을 통해 옷을 획득하는" 컬렉션 시스템을 추가한다. 참가자는 여러 미니게임을 클리어해 캡슐 가챠 형식으로 옷을 모으고, 모은 옷으로 기존 코디 화면에서 캐릭터를 꾸민 뒤 인쇄한다.

## 2. 대상 및 환경

- 모바일 전용 (개인 폰으로 참여, QR 코드로 접속)
- 참여자 규모: 30~50명
- 실패 없는 구조 원칙 — 퀴즈(슬롯2)를 제외한 모든 게임은 정답/실력 판정 없이 클리어 가능

## 3. 전체 사용자 흐름

1. QR 코드 스캔 → 전용 웹페이지 접속
2. 닉네임 + 비밀번호 입력 (신규면 생성, 기존이면 진행상황 복구)
3. 컬렉션 허브(6슬롯) 진입 → 자유 순서로 게임 선택/클리어
4. 6슬롯 진행에 따라 옷 획득
5. 코디 화면([GameBoard.jsx](../src/components/GameBoard.jsx)) → 보유한 옷/배경/소품으로 캐릭터 꾸미기
6. 인쇄([PrintOverlay.jsx](../src/components/PrintOverlay.jsx)) — 기존 로직 그대로

## 4. 인증

- 로그인 대신 **닉네임 + 비밀번호**로 세션 식별 (전화번호는 부담을 줄 수 있어 배제)
- 닉네임은 사실상 유니크 키

### 4-1. 비밀번호 규칙
- **숫자 4자리만** 허용
- 반복 숫자(예: `1111`) 및 쉽게 유추 가능한 연속 패턴(예: `1212`, `1234`, `4321` 등) **금지**
- 이 규칙은 검증 로직으로만 걸러내고, 입력 안내 문구에는 노출하지 않음 (사용자가 규칙을 보고 우회 패턴을 고민하게 만들지 않기 위함)

### 4-2. 닉네임 입력 흐름
- 닉네임 입력 시 서버에 존재 여부 확인
  - **존재하지 않는 닉네임**: "없는 닉네임입니다. 새로 등록하시겠습니까?" 확인/취소 모달 표시 → 확인 시 해당 닉네임+비밀번호로 신규 등록 후 허브 진입, 취소 시 입력 화면 유지
  - **이미 존재하는 닉네임**: 비밀번호 일치 시 별도 안내 메시지 없이 바로 진행상황 복구된 슬롯 화면으로 진입 (비밀번호 불일치 시에만 오류 표시)

## 5. 컬렉션 허브 (신규 화면)

- 6개 슬롯을 자유 순서로 노출 (진행 잠금 없음), 진행률 표시 (n/6)
- 미클리어 슬롯: 물음표 아이콘 / 클리어 슬롯: 획득한 옷 썸네일
- **슬롯 6은 황금 테두리 + 황금 "?"** 로 시각적으로 구분 (구매 전용 슬롯임을 암시)
- 슬롯 탭 → 해당 게임 화면 또는 안내 모달로 이동
- 모든 게임 클리어(6/6) 시 허브에 컴플리트 연출 + "코디하러 가기" CTA

## 6. 미니게임 상세 (슬롯 1~5)

옷 획득 풀: 라이토 / 탐정 / 아저씨 / 번개맨 / 하얀늑대 (5종, `costumes.js`의 6종 중 딸기 제외). 이 5종을 게임 1~5(슬롯1~5)를 통해 모두 획득하게 된다.

| 슬롯 | 게임 | 진행 방식 | 클리어 → 보상 로직 |
|---|---|---|---|
| 1 | 카드 뒤집기 | 3×4(12칸) 짝 맞추기(메모리 게임). 실제 사진 6장(각 2장, `public/card/card-1~6.jpg`) 사용 | 6쌍 모두 맞추면 클리어 → 캡슐 가챠 연출로 남은 미획득 옷 중 랜덤 1개 배정 |
| 2 | 퀴즈 | 3문항. Q1: 1년간 새로 나온 캐릭터 이름 3개 이상 (정답: 아담/탐정/하얀늑대). Q2: 닛몰캐쉬가 부산까지 걸어간 후 기부한 금액은? Q3: 미정 | **정답이어야 통과** (오답 시 재시도 가능, 진행 자체는 막히지 않음). 통과 시 가챠로 랜덤 옷 1개 |
| 3 | 룰렛 (옷 순환+정지) | 캐릭터 위에서 의상이 빠르게 순환(START) → STOP으로 정지, 정지 시점에 걸린 옷이 결과 | 결과가 이미 보유 중인 옷이면 "다시 돌리기"로 재도전, 새 옷이면 그대로 확정 클리어 |
| 4 | 캐치캐치 | 캐릭터가 옷을 입은 채로 제자리에서 빠르게 순환(START) → STOP으로 정지 시 입고 있던 옷 획득 (꽝 없음) | 결과가 이미 보유 중인 옷이면 재도전, 새 옷이면 그대로 확정 클리어 |
| 5 | 흔들기 캡슐 | 폰을 흔들면(DeviceMotion) 캡슐에 균열이 생기다 팡 터지며 보상 공개 | 흔들기 횟수 임계치 도달 시 클리어 → 가챠로 랜덤 옷 1개 |

> **개발 범위 안내**: 슬롯5(흔들기 캡슐)는 현재 **기획만 확정된 상태이며 개발은 보류**한다. 별도 지시가 있을 때까지 구현하지 않는다.

모든 게임 공통: 클리어 시 동일한 "캡슐 가챠 리빌" 연출 컴포넌트를 공유 사용.

### 6-1. 흔들기(슬롯5) 기술 참고 (개발 보류 상태, 참고용)
- iOS Safari 13+는 `DeviceMotionEvent.requestPermission()` 권한 팝업이 필요 (사용자 버튼 탭으로 트리거해야 함), HTTPS 필수
- 안드로이드는 별도 권한 팝업 없이 동작
- **권한 거부 또는 미지원 브라우저 대비**: "연속 탭하기" 버튼이 자동 노출되어 흔들기 대신 탭으로도 캡슐을 채워 열 수 있어야 함 (실패 없는 구조 유지)

## 7. 슬롯 6 — 럭키드로우 (구매 보너스)

- 보상: 딸기 의상 (게임 풀에서 완전히 제외된 구매 전용 스페셜 아이템)
- 슬롯 탭 → "럭키드로우 - 포토카드 구매시 받을 수 있습니다" 안내 모달 + **관리자 확인** 버튼
- 관리자 확인 버튼 → 관리자 비밀번호 입력 UI → **서버에서 검증** → 통과 시 슬롯6 클리어 처리(딸기 지급)
- **중요**: 유료 구매와 직결되는 보상이므로, 이 값은 반드시 관리자 인증 서버 API를 통해서만 true로 바뀌어야 하며, 일반 사용자가 호출 가능한 API로는 절대 변경 불가해야 함 (클라이언트 단독 검증 금지)

## 8. 백엔드 / DB

- 배포: Vercel, DB: Neon (서버리스 Postgres)
- 참여자 규모(30~50명)를 고려해 저장 효율보다 **가독성/유지보수 용이성**을 우선한 슬롯별 컬럼 방식 채택 (비트마스크 등 압축 인코딩은 이 규모에서 불필요한 과최적화로 판단해 배제)

```sql
players
- nickname      (고유값, 로그인 키)
- password      (해시 저장 권장)
- slot1 ~ slot6 (각 슬롯에서 획득한 옷 id, 또는 null = 미획득)
```

### 필요 API (개략)
- 로그인/신규생성: 닉네임+비밀번호 확인 또는 생성, 진행상황(slot1~6) 반환
- 슬롯 클리어 갱신: 게임 클리어 시 해당 slotN에 옷 id 기록
- 관리자 인증(슬롯6 전용): 관리자 비밀번호 검증 → 통과 시에만 slot6 갱신 (서버 전용 로직, 클라이언트 우회 불가하도록 설계)

## 9. 기존 시스템과의 연동

- **코디 화면** ([GameBoard.jsx](../src/components/GameBoard.jsx)): 옷 선택 목록에서 미보유 옷은 잠금 표시. `costumes.js`의 COSTUMES 데이터 구조 자체는 변경 불필요, 보유 여부 플래그만 얹으면 됨
- **배경/소품 선택**: 변경 없음, 기존 로직 그대로 유지
- **인쇄** ([PrintOverlay.jsx](../src/components/PrintOverlay.jsx), [savePhotoCard.js](../src/utils/savePhotoCard.js)): 변경 없음

## 10. 열린 이슈 (제작 중 확정 예정)

- 퀴즈 3번 문항 내용
- 관리자 비밀번호 입력 UI 디테일 (마스킹 등)

## 11. 제작 난이도/리스크 메모

- 이 프로젝트는 기존에 순수 프론트엔드(서버/DB 없음)였고, 이번에 Neon DB + Vercel 서버리스 함수가 처음 추가되는 구조적 전환점
- 미니게임 5종 자체 난이도는 낮음~중간이나 종류가 많아 물량이 있음 (룰렛/캐치캐치는 구조가 유사해 재사용 가능)
- 흔들기(슬롯5)는 iOS 권한 처리 등으로 실기기 테스트가 가장 필요한 항목
